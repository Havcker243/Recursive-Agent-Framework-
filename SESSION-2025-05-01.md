## Feature 1: Run Fork

Branch any completed node in the execution graph as a new independent run.

### How it works

- User clicks a completed `raf-node` in the inspector.
- A fork panel appears showing estimated agent call cost, agent count controls (consortium and jury size), and an editable goal textarea.
- On submit, the frontend calls `POST /api/run/{run_id}/fork` with the node ID and optional goal override.
- The backend (`server/run_manager.py` → `fork_run`) collects up to 5 of the shallowest ancestors from the parent run's event log, builds an enriched goal string with ancestor context and the node's prior output, then creates a new independent run.
- The new run opens as its own session in the frontend — it does not replace the active session.

## Feature 2: Goal Chaining / Pipeline

Chain multiple RAF runs sequentially so the output of step N feeds into step N+1 via `{{output}}` substitution. The pipeline runs in parallel with (never replaces) the active main run.

### Architecture

- A floating draggable panel (`PipelinePanel.tsx`) opens from the graph toolbar via a `Link2` icon button.
- Each step is a textarea with a goal. Steps run one at a time in sequence.
- The `{{output}}` placeholder in any step is replaced with the previous step's final output text before the run starts.
- Each step creates its own independent WebSocket (`waitForRunDone`) so the main session's `wsRef` is never touched.
- A `pipelineCancelRef` (`useRef(false)`) provides cooperative cancellation — the async handler checks it before updating state.

### New file: `web/src/components/PipelinePanel.tsx`

Exports:

```typescript
export type PipelineStep = { id: string; goal: string }

export type PipelineStepResult = {
  stepId: string; runId: string; runToken: string; sessionId: string
  status: "pending" | "running" | "done" | "error" | "cancelled"
  output?: string; goal: string
}

export function PipelinePanel(props: Props): JSX.Element
```

The panel uses pointer-capture drag (`setPointerCapture`) on the header. A button-click guard prevents drag capture when the user clicks the close button. `onPointerCancel` is wired to clear the drag ref when the OS interrupts a drag gesture.

### `waitForRunDone` (in `App.tsx`)

```typescript
const waitForRunDone = (stepRunId: string, stepToken: string): Promise<string | undefined> =>
  new Promise((resolve, reject) => {
    const ws = new WebSocket(wsUrl)
    ws.onmessage = (msg) => {
      if (pipelineCancelRef.current) { ws.close(); reject(new Error("cancelled")); return }
      const ev: RafEvent = JSON.parse(msg.data)
      if (ev.event === "run_done") { ws.close(); resolve((ev as any).result?.output) }
    }
    ws.onerror = (e) => { console.error("[pipeline] WebSocket error for run", stepRunId, e); reject(...) }
    ws.onclose = (e) => { if (!e.wasClean) console.error("[pipeline] WebSocket closed unexpectedly..."); reject(...) }
  })
```

### `runPipeline` flow

1. Set `pipelineCancelRef.current = false`, `setPipelineRunning(true)`
2. For each step: substitute `{{output}}`, POST to `/api/run`, call `waitForRunDone`
3. On step completion: mark result `done`, store output for next step's substitution
4. On cancel (`pipelineCancelRef.current === true`): mark remaining steps `cancelled`
5. On error: mark step `error`, stop pipeline

### `viewPipelineStep`

Fetches `/api/run/{runId}/events` and `/api/run/{runId}` for a completed step, replays events into the main viewport as a read-only `[pipeline:N]` session without opening a new live WebSocket.

## Feature 3: Node Replay

Re-run a single completed node as a fresh isolated execution in the background. The current session graph stays visible and untouched.

### Replay vs Fork

| | Replay | Fork |
|---|---|---|
| Ancestor context | None (clean run) | Root → parent context included |
| Prior output | None | Included in prompt |
| Can have children | No (1 node cap) | Yes |
| Switches session | No | Yes — opens new session |
| Use case | Test a better answer for this exact node | Branch with a different goal or approach |

### `replayNode(nodeId, nodeGoal)` (in `App.tsx`)

```typescript
const replayNode = async (nodeId: string, nodeGoal: string) => {
  const capturedNodeId = nodeId
  replayForNodeIdRef.current = capturedNodeId
  setReplayForNodeId(capturedNodeId)
  setReplayStatus("running")
  // POST /api/run with max_nodes_total: 1, skip_clarify: true
  // Open isolated WebSocket — guard all state updates with:
  if (replayForNodeIdRef.current !== capturedNodeId) return
}
```

The `replayForNodeIdRef` ref (not state) guards the async WebSocket callback. If the user clicks a different node before the replay finishes, the callback sees the ref has changed and silently discards its result without corrupting the new node's display.

### Replay panel in the node inspector

Shown on all completed nodes (both success and failure):

- **Failed node:** amber border, "Retry this node" label
- **Successful node:** neutral border, "Replay this node" label
- Agent count steppers (consortium 1–6, jury 1–4) shown when idle
- Status badge: Running… / ✓ Done / ✕ Failed
- Output preview (200 chars) when done
- "View full result" loads the replay run into the main viewport

## Error Handling Audit

All errors are logged to structured channels only — nothing raw is shown in the UI.

### Frontend (`console.error` / `console.warn` with module prefixes)

| Module prefix | Where |
|---|---|
| `[ws]` | Main WebSocket message parse errors |
| `[sessions]` | localStorage read/write failures |
| `[models]` | `/api/models` fetch failures |
| `[runs]` | Server run list refresh failures |
| `[run]` | `startRun`, `cancelRun` failures |
| `[fork]` | `forkRun` failures |
| `[replay]` | `replayNode`, `viewReplayResult`, `fetchFreshExportData`, clipboard |
| `[plan]` | `approvePlan` failures |
| `[pipeline]` | All pipeline functions (WebSocket, run creation, event fetch) |

### Backend (`logging` module)

```python
logger = logging.getLogger(__name__)   # in both main.py and run_manager.py
```

| Location | What is logged |
|---|---|
| `run_manager._execute` | `logger.exception("Run %s failed: %s", ...)` on any unhandled exception |
| `run_manager.fork_run` | `logger.warning(...)` when `node_id` not found in parent events |
| `run_manager` fallback adapter | `logger.debug(...)` instead of silent `except Exception: pass` |
| `main.py` dotenv import | `logger.debug("python-dotenv not installed...")` |
| `main.py` WebSocket disconnect | `logger.debug("WebSocket client disconnected from run %s", ...)` |

## Design Decisions and Rationale

**Pipeline parallel to main run.** The pipeline uses completely separate state (`pipelineResults`, `pipelineRunning`) and its own WebSocket per step. The active session's `runId`, `graphNodes`, `events`, and `wsRef` are never touched. This means a user can watch their main run's graph evolve while a pipeline runs alongside it.

**`replayForNodeIdRef` instead of state.** Async WebSocket callbacks run after the component may have re-rendered with a different `selectedNode`. A `useRef` persists its value across renders and can be checked synchronously inside the callback, preventing stale closures from corrupting the wrong node's panel.

**`waitForRunDone` `settled` guard.** The `settled` boolean inside `waitForRunDone` ensures that `resolve` or `reject` is called exactly once even if the WebSocket fires both `onclose` and `onmessage` close-path handlers in the same tick.

**Fork panel highlight over navigation.** Rather than automatically switching the inspector to the fork panel (which would collapse the replay result the user just reviewed), "Fork from here" keeps the user on the replay panel and brings the fork panel into view with a visual ring. The user completes the fork intentionally.

**`pipelineCancelRef` is a ref, not state.** Setting state is asynchronous — `pipelineCancelRef.current = true` takes effect immediately inside the currently-executing `runPipeline` loop and inside the `waitForRunDone` callback without waiting for a re-render.
